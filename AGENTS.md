# Codex向けプロジェクト指示

## 作業開始時

1. 最初に `docs/README.md` を読み、現在のフェーズ、現在のステップ、正本の所在を確認する。
2. 依頼に関係する正本文書だけを追加で読む。資料間に矛盾がある場合は、`docs/README.md` の「文書の正本」に従う。
3. 大きい変更では、実装前に「目的・関係する文脈・制約・完了条件」を短く整理する。

明示的に別の依頼がない場合の開始点は、**Phase 1 / Step 1.1 — TypeScript / Viteの最小Webアプリ、カメラ許可、計測画面を作る**である。現在地は必ず `docs/README.md` を正とし、このファイルへフェーズ詳細を重複記載しない。

## プロジェクトの優先原則

- まずTechnical Labで二手追跡、時刻、単体ジェスチャー、計測ログを検証する。90秒MVPや演出を先に作り込まない。
- 日付、週数、作業時間ではなく、成果物と出口条件でフェーズを進める。
- 普通のスマートフォンのインカメで、音とのずれが不快でない身体的な音楽ゲーム体験を成立させることを優先する。
- プレイヤーを罰しない。追跡喪失など機械側の失敗をプレイヤーのMISSとして扱わない。
- キャラクター、大画面、立位、二人同時プレイ、配信向けの大型演出は重要な将来像だが、現行POC/MVPへ混ぜない。
- Webを第一候補とするが、実測がゲートを満たさない場合だけPCまたはネイティブ比較へ進む。

## 文書と言語

- 企画、POC/MVP、ロードマップ、UI/UX、テスト設計、意思決定の本文は日本語で維持する。
- コード識別子、API名、ライブラリ名、設定キー、コマンド、エラー文は英語の原文を維持する。
- 日本語文書を英語へ全面翻訳しない。英語話者が参加した時点で、必要なら英語の概要版を追加する。
- 同じ仕様を複数文書へ複製しない。正本を更新し、他文書からはリンクする。
- 確定仕様、仮説、条件付き分岐、将来構想を明確に分ける。
- 現在の製品仕様、ライブラリ、ブラウザ対応、ライセンス等を調査するときは、公式ドキュメントや標準仕様などの一次ソースを優先し、確認日と参照先を残す。資料にない結論は推論だと明示する。

## 実装と検証

- 依存関係とモデルは検証済みバージョンへ固定し、`@latest` を使わない。
- カメラ処理はlatest-frame-onlyを守り、古いフレームを待ち行列へ溜めない。
- ゲーム判定は音声タイムラインとイベント時刻を基準にし、描画フレーム数へ依存させない。
- 追跡バックエンドとゲーム／ジェスチャー判定を分離する。
- 生カメラ映像は原則保存しない。派生ランドマークや計測値を優先し、映像保存が必要なら事前に明示する。
- 変更に応じて、関連するテスト、型検査、lint、ビルド、実機確認のうち実行可能なものを行う。実行できない検証は未実施理由を報告する。
- UIを変更した場合は、対象画面を実際に表示して見た目と操作を確認する。

## 完了条件と引き継ぎ

作業を完了とする前に、次を確認する。

- 依頼または正本文書の出口条件を満たしている。
- 関連する検証を実行し、結果を確認している。
- 仕様やフェーズが変わった場合、先に正本を更新し、`docs/README.md` の現在地も同期している。
- `git diff` と `git status` を確認し、無関係なユーザー変更へ触れていない。
- 最終報告に、変更内容、検証結果、残る未決事項または次の一手を簡潔に記す。

## Git記録・公開の承認

- 変更規模にかかわらず、Codexは作業完了時に自動でcommitまたはpushしない。
- 特に大きな調整では、実装と検証が完了しても作業ツリーへ未コミットのまま残し、差分と検証結果をユーザーへ報告する。
- commit、pushは、その時点の依頼でユーザーが明示した場合にだけ行う。過去の「全部実施」「今後は上げてよい」等の包括的な許可を、後続作業の恒常的な承認として扱わない。
- ユーザーがcommitだけを依頼した場合はpushしない。pushや公開・デプロイも、それぞれ明示的な依頼がある場合に限る。
- commitする場合は、対象差分を確認し、検証可能な一まとまりを意図的に記録する。
