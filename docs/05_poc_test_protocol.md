# 空間ジェスチャー音楽ゲーム POCテスト手順

- 更新日: 2026-07-20
- 文書種別: Phase 1 / Phase 2の実施手順・記録・合否定義
- ステータス: v0.1（初回実施前）
- 正本範囲: POCテストの条件、分類、記録、ゲート判定

## 1. 目的

このテストは市場性を証明するものではない。次の二点を、同じ条件で繰り返し確認する。

1. 普通のスマートフォンのインカメで、二手と3ジェスチャーを再現可能に認識できるか。
2. 音との反応差が、音楽ゲームとして不快な水準になっていないか。

## 2. 対象とデータ方針

- 初期テスター: 制作者本人、ターゲット属性に近い1名
- 端末: iPhone 15、Google Pixel 10 Pro XL
- 主な組み合わせ: ターゲットテスター／iPhone 15、制作者／Google Pixel 10 Pro XL
- 端末を入れ替えるクロス試験: 初回ゲートの必須条件ではない。OS差と個人差を分ける必要が出た場合に追加する
- 生映像・生音声: 原則保存しない
- 保存対象: 派生ランドマーク、イベント、時刻、性能値、回答、画面録画（同意がある場合のみ）

10代の参加者については、参加者本人へ目的、保存項目、共有範囲、削除方法を平易に説明する。年齢、地域、テストの実施形態に応じて必要な保護者確認を行う。画面録画やデバッグ映像を第三者へ共有する場合は、テスト参加への同意とは別に確認する。

## 3. 固定する試験条件

| 項目 | 初期条件 |
|---|---|
| 姿勢 | 椅子に座る |
| 端末 | スタンドへ横置き |
| 距離 | カメラから70〜100cm。実測値を記録 |
| カメラ | インカメ |
| 音声 | 本体スピーカー。Bluetooth禁止 |
| 照明 | 顔と手に正面または斜め前から光が当たる一般室内 |
| 背景 | 手と極端に同化しない背景。柄・逆光の有無を記録 |
| 服装 | 長袖／半袖、袖色を記録。試行中は変更しない |
| ブラウザ | ブラウザ名と完全なバージョンを記録 |
| OS | OS名と完全なバージョンを記録 |
| 音量 | 会話より少し大きい一定値。端末の音量段階を記録 |

条件を変えた試験は同じ結果表へ混ぜず、別セッションとして記録する。

## 4. テスト前チェック

1. 端末、OS、ブラウザ、モデル／アプリ版を記録する。
2. Bluetoothが切れていることを確認する。
3. 端末が横向きで安定し、プレイヤーが画面を無理なく見られることを確認する。
4. 左右の手を胸〜肩の高さで広げ、両方が快適領域へ収まることを確認する。
5. 生映像を保存しない設定を確認する。
6. 開発ログの時刻、端末時刻、試行番号を確認する。
7. テスターへ「成功させることより、感じたずれや誤認識をそのまま伝えることが重要」と説明する。

## 5. Phase 1 — 制御試験

各ジェスチャーを説明・実演した後、次の順で行う。左右・方向の偏りを避けるため、全テスターで同じ順序を使う。

各試行は、カウント中の準備期間とrecognition windowを分ける。音声targetがある場合は`targetTimeMs - 500ms`からイベントと拒否理由を採用し、deadlineは`targetTimeMs + 30,000ms`とする。音声targetがない場合は試行開始時刻をwindow開始とし、そこから30秒をdeadlineとする。準備期間の開始位置合わせや拒否は試行結果へ加えない。

期待ジェスチャーが成立しない場合も試験を停止し続けない。根拠を確認できる場合は手動分類し、判断できない場合は「未成立として次へ」を選ぶ。deadlineへ到達した試行は自動で次へ進む。手動スキップとtimeoutはいずれも`unclassified`として完了件数と各ジェスチャー10回の分母へ含め、解決経路をそれぞれ`manual-skip`、`trial-timeout`として記録する。timeoutだけを根拠に`player miss`または`machine miss`へ分類しない。

timeoutまたは手動スキップ後は約1秒だけ結果を表示し、次の試行を新しいtargetで準備する。ページ非表示中に試行を連続消化せず、復帰時に期限を過ぎた現在の1試行だけをtimeoutとして解決する。

### 5.1 エアタップ 10回

- 左側5回、右側5回を交互に行う。
- ターゲット面を通過し、止めたままにしない。
- 各試行の間にクールダウン表示を待つ。

### 5.2 リボンスワイプ 10回

- 左→右、右→左を各3回。
- 左下→右上、右下→左上を各2回。
- 指定方向、中心通過、成立時刻を別々に記録する。

### 5.3 クラップ／ニアクラップ 10回

- 実接触5回、中央の光球を挟むニアクラップ5回。
- 強く叩かない。音量や衝撃を評価しない。
- 接触フレームが消えても、収束軌跡と推定接触時刻を記録する。

### 5.4 Phase 1の記録

- 成功回数／10
- player miss
- machine miss
- false trigger
- tracking loss
- eventTimeと目標時刻の差
- 推論p50 / p95、追跡出力Hz、frame age p95
- ID swapと理由コード
- 各試行の準備、window開始、target、deadline、完了時刻
- 解決経路（gesture event、手動分類、手動スキップ、trial timeout）
- 試行別の拒否理由とtracking gap

## 6. Phase 2 — Interaction POC試験

短い統合シーケンスを、各テスターが同じ端末条件で3回行う。

1. 初回は画面内チュートリアルだけを使い、口頭で正解を教えない。
2. 1回目終了後、操作方法ではなく感想だけを聞く。
3. 2回目と3回目は自由にリトライする。
4. 各回の間に設定や閾値を変えない。変更が必要なら別セッションとする。
5. 各回終了直後に同期感、疲労、痛み、分かりにくかった動作を記録する。

## 7. Phase 3 — M1-Web MVP品質試験

90秒MVPが完成した後、5人テストへ出す前に実施する。

1. 各ジェスチャーを各端末で20回以上行い、controlled recall 95%以上を確認する。
2. 90秒MVPを同じ設定で繰り返し、連続10分以上動作させる。
3. 10分中のfalse trigger、二手coverage、追跡Hz、frame age p95を集計する。
4. 開始時と終了時のメモリ、フレーム待ち行列、推論p50 / p95を比較する。
5. 10分後も入力品質が階層を跨いで低下しないことを確認する。
6. 一項目でも満たさない場合は、M1をPassにせず、Learnとして原因と次の変更を一つ記録する。

## 8. 分類ルール

| 分類 | 定義 | 判定方法 |
|---|---|---|
| success | 期待ジェスチャーが成立し、イベント時刻が記録された | イベントログ |
| player miss | 指定方向、位置、順序、時刻等をプレイヤーが満たしていない | ランドマーク再生＋譜面条件 |
| machine miss | 参照条件を満たす軌跡があるのに正しいイベントが出ない | ランドマーク再生＋reasonCodes |
| false trigger | 対象操作をしていないのにイベントが出る | イベント前後の派生軌跡 |
| tracking loss | 必要な手が追跡できず判定不能 | trackingConfidenceと欠落時間 |
| unclassified | ログだけではplayer／machineを区別できない | 推測でどちらかに寄せない |

`unclassified`を除外して成功率を良く見せない。手動スキップとtrial timeoutも各ジェスチャー10回の分母へ含め、件数、解決経路、原因をそのまま残す。成功、手動分類、手動スキップ、timeoutが競合した場合は、最初に確定した結果だけを一件記録する。

## 9. 主観質問

各プレイ後に、誘導せず次の順で聞く。

1. 音と動きの合い方を1〜5で評価すると？
2. 遅い、早い、ばらつく、のどれかを感じた？
3. 何をすべきか分からなかった場面は？
4. 分かって動いたのに、ゲームに無視されたと感じた場面は？
5. 一番印象に残った動作は？
6. もう一度遊びたい？　そう思った理由は？
7. 肩、手首、目、頭の疲れをそれぞれ0〜10で評価すると？
8. 痛みや不快感はあった？

## 10. ゲート対応表

| ゲート | 目的 | 合格条件 | 次の行動 |
|---|---|---|---|
| P1-Controlled | 単体認識が成立するか | 2つの主なテスター／端末組み合わせで各ジェスチャー8/10以上を出発点とする | Phase 2へ統合、または状態機械・画角を修正 |
| P2-Interaction | 短い体験が再現可能か | false trigger 90秒に1回以下、machine miss 90秒に2回以下、同期感4/5以上、3回目も悪化しない | Phase 3のMVP制作へ進む |
| M1-Web MVP | Web版を広いMVPテストへ出せるか | 二手coverage 95%以上、追跡25Hz以上、frame age p95 140ms以下、controlled recall 95%以上、false trigger 10分に1回未満、1曲中にフレーム待ち行列・メモリが増加しない | 5人テスト／端末拡張へ進む |
| R1-Ranked | 将来の競技モード候補 | 技術計画のRanked候補条件 | 製品方針で競技モードを承認した場合だけ使用 |

P1/P2は少人数POCの学習ゲート、M1はMVPを広いテストへ進めるための品質ゲートであり、同じ数値ではない。
