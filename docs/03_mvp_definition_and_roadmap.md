## 1. POCとMVPで証明すること

> 普通のスマートフォンのインカメでも、音楽とのずれが不快にならない精度で遊べ、既存のスマートフォン音楽ゲームとは異なる身体的な楽しさが生まれる。

優先順位は次のとおりとする。

1. **体験仮説**: 身体で曲を開放する操作に、新しい楽しさがある。
2. **技術ゲート**: カメラ入力の遅延と誤認識が、その楽しさを壊さない。

技術精度そのものを商品の魅力とはせず、成立に必要な合格条件として扱う。

| 成果物 | 証明すること | 完了地点 |
|---|---|---|
| Technical Lab | 二手追跡、時刻、単体ジェスチャーを測定・説明できる | Phase 1 |
| Interaction POC | 3ジェスチャーを統合しても、遅延と誤認識が操作感を壊さない | Phase 2 |
| 90秒MVP | ターゲットが新しい楽しさを理解し、自発的に再プレイする | Phase 3 |

## 2. 初期ターゲット

- 年齢: 10代中盤〜後半
- 経験: プロジェクトセカイ、あんさんぶるスターズ等のスマートフォン音楽ゲーム経験
- 期待: 初見でも理解でき、プレイ動画として動作と画面効果が伝わる
- 初期テスター: 制作者本人と、ターゲット属性に近い1名

制作者本人の評価は遅延・不具合・使いにくさの発見に使う。新規性や再プレイ意向については、ターゲットテスターの反応をより強い証拠として扱う。

## 3. POCの対象環境

- 主端末: iPhone 15、Google Pixel 10 Pro XL
- ブラウザ: 各端末の標準的な最新ブラウザ経路から検証
- 姿勢: 座位のみ
- 設置: スマートフォンスタンド必須
- 向き: 横画面を第一仮説とする
- 距離: 机上設置で70〜100cmを初期値とし、両手カバレッジから調整
- カメラ: インカメ
- 音声: 本体スピーカーのみ。BluetoothはPOC対象外

横画面を永久に固定する判断ではない。横POC成立後、縦画面を短時間比較し、両手カバレッジ、譜面の読みやすさ、録画時の見栄えで判断する。

## 4. MVPスコープ

### 含むもの

- 約90秒、1曲、Easyのみ
- Pulse / Flow / Burstを一つずつ含む
- エアタップ
- リボンスワイプ
- Crush & Burst（クラップ／ニアクラップを含む）
- SFホログラム空間の最小演出
- 認識した左右の手を示す光点・輪郭・軌跡
- アクションごとに異なる短い効果音
- 曲を止めないノーフェイル進行
- ポジティブなリザルトと即時リトライ
- 開発用のカメラ映像・ランドマーク表示切り替え

### 含まないもの

- キャラクターとバーチャル・ハイタッチ
- 曲そのものを成功・失敗で差し替える動的アレンジ
- Normal以上の難易度
- ランキング、対戦、課金、アンロック
- 複数曲、譜面エディタ、着せ替え
- 立位、全身追跡、二人同時プレイ
- Bluetooth音声の遅延対応

仮キャラクターでは感情的な関係性を正しく測れないため、キャラクターはMVP後へ送る。クラップの予告は、左右から収束する浮遊ハンドサインで表現する。

## 5. 音とフィードバック

- 楽曲本体はプレイ結果によって変えない。
- エアタップは短く明瞭な打音、リボンスワイプは軽い掃過音、Crush & Burstは圧縮音＋開放音とする。
- 効果音は拍や主旋律を邪魔せず、成功時刻を理解できる音量・帯域にする。
- ミス専用の不快な警告音を鳴らさない。
- 将来のキャラクター掛け声は、曲に同期し、楽曲を遮らない頻度と長さで検討する。

## 6. ノーフェイル方針

- スコアが低くても曲を中断しない。
- プレイヤーを失敗画面へ送らない。
- リザルトは罰ではなく、「できたこと」と次の一歩を返す。
- 技術的な追跡喪失をプレイヤーのMISSとして表示しない。

## 7. 2人で行うPOC合格基準

実施条件、分類、質問、記録様式は[05_poc_test_protocol.md](./05_poc_test_protocol.md)を正本とする。Phase 2では各テスターが担当端末で統合シーケンスを3回ずつ実施し、単発の成功ではなく再現性を見る。

### 技術ゲート

- 練習画面で各ジェスチャーが10回中8回以上成立する。
- 意図しない誤発火が90秒に1回以下。
- 正しく動いたのに無視された明確な機械ミスが90秒に2回以下。
- 2人とも音との一体感を5段階で4以上と評価する。
- 3回目も1回目と同程度の認識と遅延感を維持する。

### 体験ゲート

- ターゲットテスターが、印象に残った場面としてCrush & Burstまたはリボンスワイプを自発的に挙げる。
- 画面の予告だけで次の動作を理解できる。
- ターゲットテスターが自発的にリトライする。
- 疲労が10段階で3以下で、肩・手首の痛みがない。
- 15秒の画面録画を作成し、Phase 4の第三者理解テストに使える見せ場を残す。この項目は2人POCの合否には使わない。

2人の結果は製品市場性の証明ではない。POC成立後に5人以上のターゲットテストへ移行する。

ここで定める80%・90秒単位の条件はP1/P2の少人数学習ゲートである。技術計画の95%・10分単位の条件は、Phase 3のMVPを広いテストへ出すためのM1-Web MVP品質ゲートであり、同じ合格基準ではない。

## 8. 方向転換ルール

- 両端末で遅延が不快: カメラ処理、判定時刻、Web実装を再評価し、PC比較またはネイティブ比較へ進む。
- 一方のOSだけ不快: コンセプトを捨てず、プラットフォーム固有経路または初期対応端末の限定を検討する。
- 技術は合格するが楽しくない: 端末ではなく、譜面、効果音、予告、ジェスチャー構成を変更する。
- Crush & Burstが誤認識する: 実接触を要求せず、ニアクラップと収束・開放軌跡を中心に再設計する。

## 9. ゲート式ロードマップ

このロードマップは日付、週数、工数見積もりでは管理しない。各フェーズを「作るもの」「確認すること」「出口条件」で管理し、出口条件を満たした時点で次へ進む。AIを継続的に稼働できることを前提に、便宜的な締切や作業時間を進捗指標にしない。

運用ルール:

- 現在のフェーズと、次に満たす出口条件を常に一つ明示する。
- 完了は経過時間ではなく、成果物と検証結果で判断する。
- 速く実装できても、技術ゲートと体験ゲートを未確認のまま飛ばさない。
- 想定外の結果が出た場合は、日程を動かすのではなく、方向転換ルールに従って次のステップを変える。

### Phase 0 — 定義固定（完了）

- 対象、端末、姿勢、画面向きの第一仮説
- MVPの3ジェスチャー
- キャラクターなし、ノーフェイル、楽曲非改変
- 2人向け評価基準

### Phase 1 — Tracking & Latency Lab

- 二手トラッキング、最新フレームのみの処理
- 横画面・座位・70〜100cmのカバレッジ確認
- 音声クロックとジェスチャー時刻の統合
- エアタップ、リボンスワイプ、クラップ／ニアクラップの単体試験
- iPhone / Androidの比較ログ

出口: `05_poc_test_protocol.md`のP1-Controlledを両端末で通過する、または方向転換先を決められる。

### Phase 2 — Interaction POC

- 3ジェスチャーを一つの短いシーケンスへ統合
- アクション別効果音
- 生カメラを隠した手の可視化
- 浮遊ハンドサインとCrush & Burstの予告
- ノーフェイル進行と簡易リザルト

出口: 2人が3回ずつ再現可能にプレイし、`05_poc_test_protocol.md`のP2-Interactionを通過する、または学習結果に基づく次の変更を一つ決める。

### Phase 3 — 90秒MVP垂直スライス

- 1曲の譜面とPulse / Flow / Burst構成
- SFホログラム演出
- セットアップ、短いチュートリアル、プレイ、リザルト、リトライ
- 15秒のプレイ動画として成立する見せ場
- `06_mvp_chart_scoring_spec.md`に基づく譜面、Resonance、ポジティブなリザルト

出口: ターゲットテスターが自発的に再プレイし、印象的な動作を説明でき、M1-Web MVP品質ゲートを通過する。

### Phase 4 — ターゲット5人テスト

- 初見理解、再プレイ、疲労、誤認識、録画の伝達性を評価
- 横／縦画面の短い比較
- 中価格帯スマートフォンを追加

出口: MVP継続、ジェスチャー再設計、プラットフォーム限定のいずれかを決定する。

### Phase 5 — MVP後の選択

Phase 4の結果から一つの価値仮説を選び、複数の大型機能を同時に開始しない。

- ジェスチャー拡張: ツインヒット、サウンドレール、両手ポータル
- コンテンツ拡張: Normal / Hard、複数曲、譜面制作基盤
- 表現拡張: キャラクター品質を確保した上でのハイタッチ、配信向け演出
- 技術分岐: Web品質が不足した場合のPC／ネイティブ比較

## 10. POC用仮楽曲候補

ゲーム利用と編集の自由度を優先し、CC0音源を第一候補とする。

1. [Rhythm Garden — congusbongus](https://opengameart.org/content/rhythm-garden) — CC0、funky / electronic / dance。第一試聴候補。
2. [A Flawless Getaway — vitalezzz](https://opengameart.org/content/a-flawless-getaway) — CC0、Sci-Fi / action、ループ版あり。
3. [PM Rhythm — Some Weirdo](https://opengameart.org/content/pm-rhythm) — CC0、120 BPM。テンポ検証用の機械的な候補。

選定条件:

- 拍が聞き取りやすい。
- BPMが一定、またはテンポマップを作りやすい。
- 90秒に切り出しても導入・展開・締めが成立する。
- 効果音を重ねても主旋律と衝突しにくい。
- ライセンスと取得時点の原本を記録できる。

Mixkitの無料音楽はゲーム利用が許可されていないため除外する。Pixabayは利用・改変可能な候補もあるが、Content ID登録曲は配信テストで通知が発生し得るため第一候補にしない。

## 11. MVP後の設計レベル

| 区分 | 内容 | 扱い |
|---|---|---|
| 確定原則 | ノーフェイル、身体を罰しない、曲本体を操作結果で乱さない | MVP後も維持する |
| 条件付き候補 | ツインヒット、サウンドレール、両手ポータル、縦画面、中価格帯端末、ネイティブ化 | Phase 4の結果または技術ゲートで一つずつ開く |
| 任意研究 | 競技モード、ランキング、狭い判定窓 | 製品方針で明示承認された場合のみ着手する |
| 将来構想 | キャラクター、ハイタッチ、大画面、立位、配信演出、Performance Mode | 北極星として保持し、現行MVPへ混ぜない |

### 北極星

> 大型PC画面の中の魅力的なキャラクターと、現実のプレイヤーが一緒に踊り、周囲や配信視聴者からも「二人で楽しんでいる」ように見えるPerformance Mode。

キャラクター、二人の関係性、大画面、立位、配信演出は重要な将来像として保持するが、現行POCの合格条件へ混ぜない。
